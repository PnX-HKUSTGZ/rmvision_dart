# 飞镖视觉历史调试记录

本页根据 Git 提交历史、代码注释和现有 README 提炼。它记录可复用的“现象—根因—修复—预防”关系，不代替具体 commit diff。查看原始历史可使用：

```bash
git log --all --oneline --decorate
git log --all --grep='<关键词>'
```

## 1. YAML 轴参数被解析为布尔值

| 项目 | 内容 |
| --- | --- |
| 现象 | 融合节点启动失败，或舱门坐标轴参数报告非法 |
| 根因 | YAML 对未加引号的 `y` 等值存在布尔解析风险；轴字符串还可能丢失正负号 |
| 修复 | 将轴配置写为带引号的显式字符串，例如 `"+x"`、`"+y"`、`"+z"` |
| 验证 | 节点启动成功，日志中的轴映射符合 Livox 实际坐标系 |
| 预防 | 对枚举参数增加启动校验；配置审查时检查 YAML 类型 |
| 相关位置 | [node_params.yaml](../../src/vision_bringup/rm_vision_bringup/config/node_params.yaml)、[range_fusion_node.cpp](../../src/rm_livox_fusion/src/range_fusion_node.cpp) |

## 2. 无目标后仍保留上一帧结果

| 项目 | 内容 |
| --- | --- |
| 现象 | 绿灯移出画面后，输出仍保持最后一次有效角度或距离 |
| 根因 | 无目标路径没有重置全部输出字段，或下游融合/mux 继续使用缓存消息 |
| 修复 | 无目标时显式生成无效状态；融合和 mux 使用超时限制陈旧数据 |
| 验证 | 目标消失后在规定超时内进入未知/无效状态，重新出现后恢复正常 |
| 预防 | 为“启动时无目标、识别后丢失、目标切换、数据源超时”建立回归用例 |
| 相关位置 | [detector_node.cpp](../../src/auto_aim/light_detector/src/detector_node.cpp)、[send_mux_node.cpp](../../src/rm_livox_fusion/src/send_mux_node.cpp) |

## 3. 默认值与真实值反复跳变

| 项目 | 内容 |
| --- | --- |
| 现象 | `/Send` 或终端日志在无效哨兵值与真实距离之间频繁切换 |
| 根因 | mux 检查和缓存时序不一致，或多个层级对默认值采用不同语义 |
| 修复 | 统一来源选择和超时逻辑，避免中间层重复覆盖有效/无效字段 |
| 验证 | 固定目标和目标消失场景中输出来源唯一，状态切换与数据源时间戳一致 |
| 预防 | 将状态有效性与数值分离；串口接口变更执行[协议检查表](05_development_templates.md#串口协议变更检查表) |

## 4. 绿灯遮挡被误判为普通无目标

| 项目 | 内容 |
| --- | --- |
| 现象 | 舱门已打开但绿灯被机构遮挡时，系统只能输出未知 |
| 根因 | 早期逻辑只依赖图像是否检测到灯，无法区分遮挡和舱门未打开 |
| 修复 | 使用原始 Livox 点云检查中央舱门 ROI，引入近距离阻挡和远处开门证据，扩展 `light_detected` 状态 |
| 验证 | 开门遮挡、未完全开门、无可靠点云和正常绿灯四类样本可区分 |
| 预防 | 保留原始点云和 `door_nearest_distance`，使用连续帧确认而非单帧判断 |
| 相关位置 | [Send.msg](../../src/auto_aim/auto_aim_interfaces/msg/Send.msg)、[range_fusion_node.cpp](../../src/rm_livox_fusion/src/range_fusion_node.cpp) |

## 5. 舱门 ROI 混入固定结构

| 项目 | 内容 |
| --- | --- |
| 现象 | 门框、滑台或近处固定结构导致舱门状态误判 |
| 根因 | 舱门 ROI 过大或中心偏移，近远点证据没有与实际开口范围对齐 |
| 修复 | 将 ROI 收缩到中央开口，分别设置近点和远点证据范围及最少点数 |
| 验证 | 固定结构不再触发错误状态，真实开门仍能在连续确认帧内识别 |
| 预防 | 使用多个姿态和场地样本，不根据单帧调 ROI |

## 6. 绿灯物理半径错误导致 PnP 距离异常

| 项目 | 内容 |
| --- | --- |
| 现象 | 目标框和圆心正确，但 PnP 距离成比例偏大或偏小 |
| 根因 | PnP 使用的绿灯物理半径与实际发光圆不一致 |
| 修复 | 重新确认物理尺寸并更新对应参数；用多个已知距离验证 |
| 验证 | 多距离下 PnP 误差趋势合理，融合的 PnP 距离门限也能正确工作 |
| 预防 | 物理尺寸、镜头、相机内参和分辨率作为一组配置记录 |
| 相关位置 | [pnp_solver.cpp](../../src/auto_aim/light_detector/src/pnp_solver.cpp)、[node_params.yaml](../../src/vision_bringup/rm_vision_bringup/config/node_params.yaml) |

## 7. 更换镜头后二值图全黑

| 项目 | 内容 |
| --- | --- |
| 现象 | 更换镜头或相机配置后，原图可见绿灯但二值图没有目标 |
| 根因 | 曝光、通道响应和目标像素半径变化，旧颜色与半径阈值不再适用 |
| 修复 | 重新确认镜头档位和内参；基于样本调整颜色及半径筛选 |
| 验证 | 不同距离的真实绿灯通过，白字和反光负样本仍被排除 |
| 预防 | 不把不同镜头的参数混用；每次切换镜头执行固定回归数据集 |

## 8. 点云背景结构抢占距离中值

| 项目 | 内容 |
| --- | --- |
| 现象 | 绿灯在远处，但融合距离稳定落在近处结构上 |
| 根因 | ROI 或角度门限包含背景/前景结构，点数足够但不属于目标 |
| 修复 | 修正外参；收紧 ROI；结合 PnP 距离门限；使用中位数、MAD 和最少点数 |
| 验证 | 多段数据中目标附近点成为主体，近处结构不再抢占中值 |
| 预防 | 导出点云投影视频，与图像和 `Send` 时间轴同步检查 |
| 相关位置 | [range_fusion_node.cpp](../../src/rm_livox_fusion/src/range_fusion_node.cpp) |

## 9. 雷达距离厘米级抖动

| 项目 | 内容 |
| --- | --- |
| 现象 | 静止目标的融合距离持续小幅变化，影响电控稳定 |
| 根因 | 点云采样、ROI 点集和中位数随帧变化；只做离群点剔除不足以稳定输出 |
| 修复 | 增加距离低通滤波和死区；大幅跳变时重置滤波，避免切目标拖尾 |
| 验证 | 静止目标抖动降低，真实目标切换仍能及时响应 |
| 预防 | 同时统计平滑程度和响应时间，不能只追求输出不变 |

## 10. 相机—Livox 外参只对单段数据有效

| 项目 | 内容 |
| --- | --- |
| 现象 | 某段 rosbag 投影正确，换距离、方向或相机角色后明显偏移 |
| 根因 | 外参过拟合单一场景，或 base/outpost 使用了同一组外参 |
| 修复 | 双相机分别配置；使用多个距离和方向交叉验证 |
| 验证 | 所有验证 bag 的投影误差均在可接受范围，而非只优化一段数据 |
| 预防 | 使用[外参标定记录模板](05_development_templates.md#相机-livox-外参标定记录模板)保存适用范围和证据 |

## 11. debug 可视化导致延迟升高

| 项目 | 内容 |
| --- | --- |
| 现象 | 开启调试图或点云绘制后，检测、融合显示或串口输出出现明显延迟 |
| 根因 | 每帧绘制、图像编码、Topic 发布、RViz 和点云投影占用 CPU/IO |
| 修复 | 比赛关闭 `debug`；减少点云绘制；使用低优先级内录并赛后离线生成可视化 |
| 验证 | `/latency` 恢复，目标输出不再因资源争用抖动 |
| 预防 | 调试与比赛配置分离；任何性能对比都记录 debug 开关状态 |

## 12. 扫码枪节点在不需要时仍启动

| 项目 | 内容 |
| --- | --- |
| 现象 | 使用电控串口输入飞镖编号和偏置时，系统仍尝试打开扫码枪设备 |
| 根因 | 旧启动逻辑没有按 `dart_input_mode` 条件创建扫码枪节点 |
| 修复 | Launch 根据 `serial`/`barcode` 模式决定是否启动扫码节点 |
| 验证 | serial 模式无扫码枪依赖；barcode 模式正常发布 `/barcode/scan_profile` |
| 预防 | 对两个模式分别做无设备和有设备启动测试 |

## 13. Livox Viewer 无法发现雷达

| 项目 | 内容 |
| --- | --- |
| 现象 | Viewer 无法连接雷达，但硬件和网络看似正常 |
| 根因 | 视觉主程序或 systemd 自启动已经占用雷达连接 |
| 修复 | 确认并安全停止占用进程后再启动 Viewer，完成操作后恢复视觉服务 |
| 验证 | Viewer 可连接；恢复服务后 `/livox/lidar` 正常发布 |
| 预防 | 调试前先检查服务状态和进程，不重复启动驱动 |

## 14. 断电后 rosbag 缺少索引

| 项目 | 内容 |
| --- | --- |
| 现象 | bag 目录存在 `.db3`，但缺少或无法读取 metadata |
| 根因 | 强制下电发生在 rosbag 完成索引和 metadata 写入之前 |
| 修复 | 保留原始数据，执行 `ros2 bag reindex` 后再检查和导出 |
| 验证 | `ros2 bag info` 可读取，Topic 和时间范围合理 |
| 预防 | 当前写入包保持可恢复的 sqlite3；只压缩已经结束且检查通过的历史包 |
| 相关位置 | [dart.sh](../../dart.sh)、[extract_all_bags.sh](../../RECORD/extract_all_bags.sh) |

## 15. 导出视频损坏或缺少容器收尾

| 项目 | 内容 |
| --- | --- |
| 现象 | 导出视频无法播放、缺失容器索引或只有部分时段 |
| 根因 | 导出进程异常中止、输入 bag 未修复或输出帧率配置不匹配 |
| 修复 | 先 reindex 和检查 bag；删除不完整导出目录后重新执行；使用实际帧率 |
| 验证 | 视频可完整播放，时长与 bag 相符，标注和图像时间轴对齐 |
| 预防 | 导出使用临时目录并在成功后完成重命名；始终保留原始 bag 直到验收完成 |

## 新增调试记录的要求

新增案例必须包含：

- 出问题的 commit 和配置版本。
- 可复现输入数据或清晰的触发条件。
- 直接根因，而不是只写最终改了哪个参数。
- 修复前后证据。
- 对其他目标、镜头、相机角色和实时性的回归结果。
- 预防措施和回滚方法。

完整复盘可复制[赛后故障复盘模板](05_development_templates.md#赛后故障复盘模板)。
