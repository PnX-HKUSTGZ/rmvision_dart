# 飞镖视觉潜在问题与开发方向候选

本页记录尚未立项或尚未完成验证的候选方向。条目不等于已确认需求；采纳前必须补充负责人、优先级、测试数据和验收标准。已发生并解决的问题应记录到[历史调试记录](07_debug_records.md)。

## 维护规则

每个候选方向经历以下状态：

```text
想法 → 证据收集 → 方案评审 → 已立项/不采纳 → 开发验证 → 关闭
```

没有可复现实验或量化收益的条目，不直接进入比赛分支。

## 候选清单

| 候选方向 | 代码依据/现状 | 主要风险 | 建议验证 |
| --- | --- | --- | --- |
| 补全并版本化所有镜头外参 | [launch_params.yaml](../../src/vision_bringup/rm_vision_bringup/config/launch_params.yaml)仍包含镜头外参 TODO；双相机外参会频繁随现场调整 | 临时值被误当长期标定；更换相机或镜头后静默失效 | 使用多个距离、方向和 rosbag 交叉验证，记录相机、镜头、内参和标定日期 |
| 完善 ROS 包元数据 | 多个 `package.xml` 仍含 TODO 描述、维护者或许可证占位 | 包发布、依赖审计和交接信息不完整 | 为每个自有包确认真实维护者、描述和许可证，再运行包级检查 |
| 移除旧 `topic_recorder` 遗留 | README 和配置明确标记旧 recorder 已弃用，新内录由 `dart.sh` 管理 | 新成员误开两套录制；重复消耗 CPU、IO 和磁盘 | 搜索所有引用，确认没有部署依赖后分阶段删除或迁移 |
| 统一运行时参数校验 | 当前参数分散在 Launch、YAML 和节点声明中，曾出现轴字符串被解析成布尔值 | 类型或范围错误导致节点启动失败或产生错误结果 | 增加启动时类型、枚举、范围、frame 和 Topic 校验；故意注入非法配置测试 |
| 生成“最终生效配置”报告 | 双相机模式下部分参数由 Launch 覆盖 YAML 默认值 | 调参者修改错误文件，难以复现实验 | 启动时输出每个角色的关键最终参数，并允许导出配置快照 |
| 规范无效值和状态语义 | 历史上角度、距离和灯状态使用过多种哨兵值，融合层也可能覆盖上游结果 | 电控难以区分无目标、传感器故障和回退值 | 定义单一接口规范；考虑增加显式有效位或状态枚举；完成新旧协议兼容评审 |
| 规范 PnP 回退策略 | 当前回退受点云有效性、距离范围和配置共同影响 | 特定赛场距离可用，换场景后可能错误回退或掩盖雷达故障 | 统计每次回退原因；用多个距离和传感器故障样本验证；输出回退计数 |
| 建立 rosbag 自动回归 | 仓库未发现针对检测、融合、目标切换和串口字段的自动测试集 | 修复一个场景时可能破坏另一个场景；比赛前依赖人工观察 | 建立匿名化小型 bag 集，自动计算检测率、融合有效率、状态跳变和延迟 |
| 增加链路健康指标 | 当前主要依靠日志、调试图和 `/latency` 判断状态 | 赛场故障定位慢，无法快速区分图像、点云、TF、融合或串口异常 | 发布低频诊断信息：帧率、点云年龄、TF 失败、ROI 点数、回退次数和串口错误数 |
| 串口协议版本与兼容性测试 | [packet.hpp](../../src/rm_serial_driver/include/packet.hpp)使用 packed 结构体直接形成载荷 | 双方字段变化时可能发生静默错位；缺少版本握手 | 增加协议版本、结构体静态断言和字节级测试向量；与电控联合测试 |
| 使用稳定设备命名 | 当前配置使用 `/dev/ttyACM*`、`/dev/ttyUSB*` | 插拔顺序变化导致串口和扫码枪设备互换 | 为正式部署编写 udev 规则，使用稳定别名并保留诊断命令 |
| 完善 systemd 部署脚本 | [dart.service](../../dart.service)包含具体用户与路径示例 | 复制到其他机器后路径错误；服务环境与交互 shell 不一致 | 将用户、工作区和日志策略参数化；测试开机、异常重启和停止流程 |
| 内录自检与完整性报告 | 新内录支持 reindex、空间清理和历史压缩，但依赖多个后台任务 | 空间不足或后台失败可能直到赛后才发现 | 启动后检查保存目录、录制进程和 metadata；停止后自动执行可读性检查 |
| 调试可视化限频 | 调试图和点云投影可能显著增加延迟 | 调车时开启 debug 改变了被观察系统的实时性 | 为图像和点云绘制增加独立限频与采样参数，对比开关前后 `/latency` |
| 舱门状态数据集与混淆矩阵 | 舱门判断依赖 ROI、近远点证据和连续帧 | 不同门框、遮挡和雷达噪声下可能误判 | 建立开门、未开、遮挡、无人和点云超时数据集，统计四种状态混淆矩阵 |
| 双相机故障降级策略 | 当前结构支持单/双相机和目标门控，但比赛中的自动降级边界未形成统一规范 | 单路相机断开后可能影响容器、目标选择或最终输出 | 故障注入：断开一路相机、停止一路 Topic、切换目标；定义允许的降级行为 |

## 建议优先收集的数据

在决定开发优先级前，先通过现有内录统计：

- 每路相机实际帧率和丢帧率。
- 绿灯检测成功率、连续丢失帧和误检类型。
- ROI 点数分布、MAD 剔除比例和融合有效率。
- PnP 回退次数及触发原因。
- `light_detected` 状态分布与错误样本。
- `/latency` 的中位数、长尾和 debug 开关差异。
- `target_id` 切换到有效输出的时间。
- 串口 CRC 错误、断连和重连次数。

## 立项模板

采用某一候选方向前，复制[功能开发设计记录](05_development_templates.md#功能开发设计记录)，至少补充：

- 可复现输入数据。
- 基线指标。
- 预期收益。
- 对实时性和接口兼容性的风险。
- rosbag 回归和真机验收方式。
- 比赛分支回滚方案。
